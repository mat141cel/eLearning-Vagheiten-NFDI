Was sind Unsicherheiten?

Computersysteme basieren auf Nullen und Einsen und verarbeiten Informationen als Richtig (1) und Falsch (0). Unsicherheiten haben in diesem Binärsystem keinen Platz. In der Archäologie und Geschichte sind Unsicherheiten allerdings omnipräsent. Diese in Daten darzustellen, ist nicht trivial.

HinweisGrundwissen Binärsystem

Binäre Zahlen (Integer)

Jeder Computer arbeitet mit Bits (0 oder 1). Eine Gruppe von 8 Bits bildet ein Byte. Zahlen werden als Binärdarstellung gespeichert: 0000 0000 ist 0 im Dezimalsystem, 0000 0001 bedeutet 1, 0000 0010 bedeutet 2, 0000 0011 bedeutet 3 und so weiter. Für positive und negative Ganzzahlen wird meist das Zweier‑Komplement‑Verfahren verwendet – das erste Bit dient als Vorzeichen (0 = positiv, 1 = negativ). So kann ein 8‑Bit‑Byte Werte von –128 bis +127 darstellen. ### Texte (Strings) Ein Text wird nicht als ein einziger Wert, sondern als Sequenz von Zeichen gespeichert. Jedes Zeichen bekommt einen numerischen Code. Alle modernen Computer verwenden die gleiche numerische Entsprechung für Zeichen, Unicode genannt (UTF‑8, UTF‑16, …). Unicode ist ein universelles System, das fast alle Schriftsysteme der Welt abdeckt. In UTF‑8 wird jedes Zeichen in 1–4 Bytes codiert. Ein String wie “Hallo” wird also als Reihe von Bytes gespeichert: 48 (H) 61 (A) 6C (L) 6C (L) 6F (O).

Alle Darstellungsmöglichkeiten von Unsicherheiten haben gemeinsam, dass sie neben dem eigentlichen Wert einen bestimmten Unsicherheitswert anbieten. Wo sich diese Markierungen für Unsicherheit befinden, erschließt sich jedoch häufig nicht unmittelbar. Außerdem belibt häufig offen, wie Nutzende diese Unsicherheit interpretieren sollen. Der Umgang mit diesen zentralen Problemen wird in diesem Modul diskutiert.

Übung 1: Darstellungsmöglichkeiten von unsicheren Datierungen.

Stellen Sie sich vor, sie wollen bei einem Hortfund verstehen, in welchen Jahren diese geprägt wurden. Liegen diese nah beieinander, oder handelt es sich um größere Zeiträume. Diese Information erlaubt Rückschlüsse darüber, welche Münzen zu diesem Zeitpunkt in Umlauf waren.

Quellen für diese Übung, sind alle Münzen, die sich zu einem römischen Kaiser Ihrer Wahl im Portal OCRE befinden. Von diesen Münzen wird die Datierungsangabe heruntergeladen, die als Zeitraum von Start- bis Enddatum (beides in Jahren) angegeben ist. Hierbei handelt es sich also um eine vage Angabe, die nicht ohne weiteres auf einem Zeitstrahl (Histogramm) dargestellt werden kann.

Denkbar sind die folgenden Möglichkeiten, diese Daten zu interpretieren und zu visualisieren:

  • Start‑ / Endjahr: jede Münze wird als Punkt des ersten / letzten möglichen Prägejahrs angezeigt.
  • Mittelpunkt: jede Münze wird als Punkt im mittleren Jahr des angegebenen Zeitraums dargestellt.
  • Gleichverteilte Gewichtung: der Zeitraum wird auf alle enthaltenen Jahre verteilt; jeder Jahr‑Anteil erhält einen Bruchteil (z. B. 1 / Länge des Zeitraums).

import numpy as np
import matplotlib.pyplot as plt
from pyodide.http import pyfetch
import pandas as pd
import urllib.parse
from IPython.display import HTML
import js

async def fetch_coin_data(url):
    """Fetch SPARQL data from Nomisma and return as DataFrame."""
    encoded_query = urllib.parse.quote(url)

    # Step 5: Build the full Nomisma API URL
    api_url = f"http://nomisma.org/query?query={encoded_query}&output=json"

    res = await pyfetch(api_url, method="GET")
    if res.status != 200:
        return None, f"Fehler beim Abrufen der Daten: HTTP {res.status}"
    
    data = await res.json()
    rows = []
    for binding in data['results']['bindings']:
        rows.append({
            'coin': binding['coin']['value'],
            'start_date': binding['start_date']['value'],
            'end_date': binding['end_date']['value']
        })

    # Create the DataFrame
    df = pd.DataFrame(rows)

    # Optional: Convert dates to numeric if you want to do math on them
    df['start_date'] = pd.to_numeric(df['start_date'])
    df['end_date'] = pd.to_numeric(df['end_date'])
    return df

def chrono_plot(data):

    years = np.arange(data.start_date.min(), data.end_date.max() + 1)

    fig, axes = plt.subplots(2, 2, figsize=(12, 8))

    # --- Chronologie nach Startdatum ---
    axes[0,0].hist(data.start_date, bins=20)
    axes[0,0].set_title("Chronologie nach Startdatum")
    axes[0,0].set_xlabel("Jahr")
    axes[0,0].set_ylabel("Münzen")

    # --- Chronologie nach Enddatum ---
    axes[0,1].hist(data.end_date, bins=20)
    axes[0,1].set_title("Chronologie nach Enddatum")
    axes[0,1].set_xlabel("Jahr")
    axes[0,1].set_ylabel("Münzen")

    # --- Chronologie nach Mittelpunkt ---
    mid = (data.start_date + data.end_date) / 2
    axes[1,0].hist(mid, bins=20)
    axes[1,0].set_title("Chronologie nach Mittelpunkt")
    axes[1,0].set_xlabel("Jahr")
    axes[1,0].set_ylabel("Münzen")

    # --- Chronologie nach gleichverteilter Wahrscheinlichkeit ---
    prob = np.zeros(len(years))
    for _, r in data.iterrows():
        span = r.end_date - r.start_date + 1
        mask = (years >= r.start_date) & (years <= r.end_date)
        prob[mask] += 1 / span

    axes[1,1].plot(years, prob)
    axes[1,1].set_title("Chronologie nach gleichverteilter Wahrscheinlichkeit")
    axes[1,1].set_xlabel("Jahr")
    axes[1,1].set_ylabel("Erwartete Münzen")

    fig.tight_layout()
    return fig

query = f"""
PREFIX nm: <http://nomisma.org/id/>
PREFIX nmo: <http://nomisma.org/ontology#>

SELECT ?coin ?end_date ?start_date WHERE {{
?type nmo:hasAuthority nm:{emperor} .
?type nmo:hasEndDate ?end_date .
?type nmo:hasStartDate ?start_date .
?coin nmo:hasTypeSeriesItem ?type .
}}
"""

data = await fetch_coin_data(query)
# Generate plot
fig = chrono_plot(data)

Wie Sie sehen, kann sich die Form des Histogramms je nach gewählter Visualisierungsmethode deutlich verändern. Sichtbare Häufungen oder Lücken können daher auch durch die Interpretation der Datierungsintervalle entstehen und spiegeln nicht unbedingt tatsächliche Veränderungen in der Münzprägung wider.

Bei all diesen Darstellungsformen besteht das Problem, dass Sie künstlich Sicherheit suggeriert. Wir wissen aber nicht, wie häufig die Münzen im angenommenen Zeitraum geprägt wurden. Wenn zwei Jahre als Prägezeitpunkte angegeben sind, kann es sein, dass die Münze nur noch im Januar des zweiten Jahr geprägt wurde. Die Wahrscheinlichkeit, dass die Münze demnach im ersten Jahr geprägt wurde ist deutlich höher, als es die Grafik zeigt.

Methode Vorteile Nachteile
Start- /Endjahr - leicht verständlich
- Klare Zuordnung		 - künstliche Häufungen am Beginn/Ende von Zeiträumen
- Ignoriert die Länge der Datierung
- künstliche Präzision
Mittelpunkt - Berücksichtigt die Länge des Intervalls
- Häufig eine pragmatische Annäherung		 - Bleibt eine künstliche Präzisierung
- Lange Zeiträume werden auf einen einzigen Punkt reduziert
- erzeugt künstliche Häufungen in der Mitte von Zeiträumen
Verteilung über den gesamten Zeitraum - Spiegelt die Unsicherheit der Datierung besser wider
- Vermeidet künstliche Punktkonzentrationen
- Methodisch komplexer
- Werte wirken künstlich „glatter“
- Anzahl der Münzen pro Jahr ist keine ganze Zahl ist schwerer interpretierbar

Wie Sie sehen, sind Auswertungen mit unsicheren Münzdaten schwerer generalisierbar. In den Folgenden Modulseiten lernen Sie, wie sie trotzdem ansprechende und belastbare Auswertungen erstellen können, selbst wenn die Datengrundlage noch unsicherer ist, wie im obigen Beispiel.
